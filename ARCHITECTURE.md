# ARCHITECTURE — 開卷有益 上架架構規格（local-first ＋ 端到端加密）

> 2026-07-05 定稿。目標：上架 Cloudflare，同時守住六條產品承諾。
> 原則：本機優先（local-first）——本機是唯一真相來源，雲端只是加密備份；
> 沒有帳號、沒有網路，產品照常完整運作。

## 0. 現況

純靜態站（Cloudflare Pages 可直接部署）：一套引擎（`web/run.js` 等）＋
資料檔（`data/<exam>/bank.json`／`explanations.json`／`figures/`）。
進度存 localStorage、不追蹤、免登入。弱點：進度脆弱（清瀏覽器＝歸零）、
無跨裝置、疑義回報無處可去。題組／承上題的情境引入規則見
`docs/adr/0002-question-group-carry-context.md`。

## 1. 六條產品承諾 → 架構決策

| # | 承諾 | 決策 |
|---|---|---|
| 1 | 完全本機、離線可用 | PWA（manifest＋Service Worker）＋ IndexedDB |
| 2 | 雲端備份加密、團隊不可讀但可維護 | 瀏覽器端 E2E 信封加密 → R2；版本控制救回；疑義走「使用者主動分享單題」明文管道 |
| 3 | 跨裝置同步 | 每筆記錄 last-write-wins（時間戳），密文增量同步 |
| 4 | App 與網頁同步 | PWA 即 App（可安裝），同一份程式碼＋同一支同步 API |
| 5 | 不餵 AI／不訓練／不賣個資 | 資料是密文——技術上不可能，而非僅承諾；無第三方分析；AI 詳解只碰公開考題，永不碰使用者資料 |
| 6 | 所有權在使用者：可匯出、可刪 | 本機一鍵匯出 JSON（離線可用）；刪帳號＝刪密文＋刪 wrapped DEK（加密粉碎） |

## 2. 加密設計（需求 2、5、6 的核心）

- **信封加密**：瀏覽器產隨機 DEK（WebCrypto AES-GCM-256）加密進度 → 只上傳密文。
- **金鑰派生**：使用者密碼 → Argon2id → KEK → 包住 DEK。伺服器只存 wrapped DEK，
  永遠看不到密碼、KEK、明文。
- **復原碼**：註冊時發一次性復原碼（使用者自存），獨立再包一份 DEK。
  密碼或復原碼擇一可解；團隊兩者皆無。
- **救回（不可讀但可維護）**：
  - 進度損毀 → R2 物件版本控制，團隊還你「先前的密文版本」，你自己解開。
  - 疑義回報 → App 打包該題（qid＋作答＋描述），經使用者明確同意才送出明文。
    主資料永遠 E2E；客服只拿到使用者選擇分享的那一小塊。
- **刪帳號＝加密粉碎**：刪 wrapped DEK 後，所有殘留備份密文永久不可解。
- **誠實的取捨**：密碼與復原碼都遺失 → 無人可救。這是真隱私的價碼，
  註冊流程必須白紙黑字告知。不設「團隊持有的後門金鑰」。

## 3. 同步模型（需求 3、4）

- 進度＝單調累積的作答記錄（qid、結果、時間戳）→ 每筆 LWW 即收斂，
  不需 CRDT（YAGNI；出現真衝突需求再升級）。
- 流程：本機 IndexedDB → 加密 → push R2 → pull 他裝置密文 → 本機解密合併。
- 伺服器僅見中繼資料（記錄 id／版本／時間戳），不見內容。
- MVP 允許「整包 LWW＋衝突提示」，之後升級每筆。

## 4. Cloudflare 服務對應

| 服務 | 用途 |
|---|---|
| Pages | PWA 靜態站（現況即可部署） |
| Workers / Pages Functions | 同步 API、帳號端點 |
| D1 | 帳號（email、wrapped DEK ×2）、同步中繼——無任何明文內容 |
| R2（開版本控制） | 進度密文＋歷史版本（救回）；零 egress 費 |
| KV（選） | session token、限流 |

登入：首選 passkey（WebAuthn，無密碼外洩面、最配 E2E）；備選 magic-link。
不接第三方 OAuth（多一個資料節點，牴觸需求 5 的極簡）。

## 5. 分階段（每階段獨立可上線）

| 階段 | 內容 | 解 |
|---|---|---|
| **P0** | PWA：manifest＋SW（shell precache＋資料 runtime cache） | 需求 1（離線／可安裝＝App）|
| P1 | 匯出／匯入 JSON、本機清除 | 需求 6 本機半 |
| P2 | passkey 登入＋E2E 密文備份（R2＋D1）＋IndexedDB 遷移 | 需求 2 備份、6 雲端刪 |
| P3 | 跨裝置增量同步（每筆 LWW） | 需求 3、4 |
| P4 | R2 版本救回＋復原碼＋疑義單題明文管道 | 需求 2 維護 |

P0 以 ~5% 工力取「本機／離線／App」八成價值，且不建後端、不動隱私承諾——先上。

## 6. 紅線（每個 PR 對照）

1. 任何明文使用者資料不得離開裝置（唯一例外：P4 疑義單題、經明確同意）。
2. 不引入第三方分析／廣告 SDK。
3. AI 管線（詳解生成／紅隊）只讀公開考題資料，禁止觸碰使用者進度。
4. 匯出功能必須離線可用（資料與金鑰都在本機，不依賴伺服器活著）。
5. 新增雲端欄位前先問：這能是密文嗎？不能→是否真的必要？
