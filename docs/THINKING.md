# 思考（不實作）— T1 字體大小／無障礙、T2 離線終極方案

> 隔夜 loop 的「思考不做」項。這裡只做評估與建議，**不動程式碼**；待使用者拍板再排進下一輪。
> 評估基於統一站 `web/` 的實際盤點（2026-06-21）。

---

## T1 — 字體大中小 + 無障礙（a11y）

### 現況盤點（實測）
- **字級單位幾乎全用 rem**：`app.css` 的 `font-size` 有 91 處用 `rem`、僅 2 處用 `px`；全檔 `rem` 用量 404。
  → 代表「改 `html` 根字級，整站等比縮放」這條路**幾乎免改版面**就能成立，成本低。
- **既有 a11y 基礎不錯**：`aria-live`／`aria-atomic`（答對錯即時朗讀區 `#sr-announce`）、`aria-current`（當前分頁／考試）、`aria-label`（雷達圖）、`aria-pressed`（程度切換）、鍵盤作答（A–D／1–4／Enter）、`prefers-reduced-motion`（1 處）、語意化 HTML（header／nav／main／footer／section）。
- **缺口**：`:focus` 樣式只有 2 處（鍵盤焦點可見性不全面）、`prefers-contrast` 0 處、無字級切換 UI、未做正式對比（contrast）稽核。

### 字體大中小：建議做法（低成本）
1. `:root` 加 `--font-scale:1;`，`html{font-size:calc(100% * var(--font-scale));}`（基準 16px）。
2. 頂部加「字級：小 / 中 / 大」三鈕（或 A− / A / A＋），點擊改 `--font-scale`（如 0.9 / 1.0 / 1.15）並寫 `localStorage`（鍵 `obig_font_scale`，跨考試共用）。
3. 因字級多為 rem，整站文字隨之縮放；需順手檢查少數 `px`（2 處 font-size + 固定寬容器）是否要改 rem，避免縮放時破版。
4. 雷達圖等 SVG 用固定 viewBox，字級放大時圖不會爆版（已是向量縮放）→ 風險低。
- **估時**：約半天（含切換 UI + localStorage + 破版微調）。

### 無障礙加強：建議做法
- **鍵盤焦點可見**：補全 `:focus-visible` 外框（目前僅 2 處）到所有可互動元素（分頁、考試選擇器已加、卡片、選項按鈕）。
- **對比稽核**：跑一次 WCAG 2.2 AA 對比檢查；暖紙底配色多數應達標，但 `--ink-faint` 小字要驗。
- **`prefers-contrast: more`**：提供高對比覆寫（加深文字／邊線）。
- **動效尊重**：`prefers-reduced-motion` 擴及考試切換與卡片 hover transition。
- **建議用 `a11y-architect` agent 做一次 WCAG 2.2 audit**（Web 平臺），產出待修清單。
- **估時**：focus 補全約半天；完整 WCAG audit + 修約 1–2 天。

### 判斷
**字體大中小：該做、列下一輪近期。** 成本低（半天）、包容性高 —— 國考族群含在職、轉職、年長重考者，放大字級對長時間作答疲勞很實際。
**a11y 全面強化：值得，但分階段。** 先補 focus 可見性（半天、高 CP），WCAG 完整 audit 排稍後（用 agent 一次掃）。

## T2 — 離線終極方案（使用者已定調：寄紙本）

### 使用者定調
終極離線不是做數位離線 app，而是「**我寄紙本給使用者**」。所以問題不是「怎麼讓網站離線跑」，而是「**要不要做一個『列印友善版／PDF 匯出』，讓你能把題目印出來郵寄？**」

### 範圍與做法（若要做）
- **最省：列印友善版（`@media print` CSS）。** 加一份列印樣式：隱藏導覽／按鈕／互動元件，把選定範圍（某科某年原卷，或某科全部）排成乾淨的題目＋答案＋詳解版面；使用者用瀏覽器「列印 → 另存 PDF」即可印出郵寄。**估時約半天。**
- **進階：結構化 PDF 匯出。** 自訂版面（題目卷 + 答案卡 + 詳解分頁、頁碼、浮水印出處），用前端 PDF 套件產生。較貴、且引入相依套件，與「無外部依賴」原則有張力。**估時 1–2 天。**
- **不做：離線單檔 app／PWA。** 與「寄紙本」的定調不符，且統一站已捨棄離線單檔機制（進度匯出／含進度副本）。

### 判斷
**先做「列印友善版」（`@media print`），不做 PDF 套件、不做離線 app。** 理由：
- 直接服務「印出來郵寄」這個真實需求，成本最低（半天）、零新依賴。
- 瀏覽器內建「列印成 PDF」已能產出可郵寄檔案，不必自己排 PDF。
- 若日後需要更精緻的卷面（答案卡、浮水印），再評估結構化 PDF —— 屆時要權衡「引入 PDF 套件 vs 無外部依賴」。
