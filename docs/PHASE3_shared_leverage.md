# Phase 3 — 共用槓桿與新增考試（查證結論 + 接手 SOP）

> 隔夜 loop 的 Goal 3。原構想：「行有餘力，把最適合、共用槓桿快贏也做起來。」
> 這份文件記錄**實證查證結論**（為何完整起一科今晚不做）+ **未來新增一科的標準流程**。

## 查證結論（2026-06-21，考選部開放資料 dataset 170565）

### 1) 現有 5 科之間：實測零題重疊
- 程式比對 stem 指紋：counseling ∩ clinical = **0 題**；其餘配對僅個位數泛句假陽性。
- 原因：諮商心理師、臨床心理師、社工師、律師、會計師是**不同類科考試，各自獨立出題**。
- 推論：所謂「臨床／諮商共用變態心理學、心理衡鑑」是**科目概念**重疊，不是**題目**重疊。
- 影響：統一站「資料按考試分檔、每題隸屬單一考試」是正確設計；不需跨科去重。

### 2) 公職社工／社會行政：存在，但不能直接吃現有題庫
- 開放資料中查到公務員體系的社工科：公務人員初等考試、地方特考的「**社會工作大意**」。
- 但這與專技高考社工師的「社會工作」「社會工作直接服務」等是**不同科目名、不同難度**
  （「大意」屬初等概要），題目不共用。
- 且這些公務員科目在資料中題目級結構化未完成（`question_count: 0`），要自建
  PDF→OCR→題目/選項/答案抽取→答案對齊 的 pipeline。

### 3) 為何完整起一科今晚不做
- 完整一科 = 「抓 PDF + 結構化 + 答案對齊 + 詳解生成 + 申論範本 + 關聯」整套，
  正是現有每科耗用大量額度建成的工程。
- 隔夜剩餘額度硬做會半殘（題庫殘缺、答案未對齊），違反「品質優先、誠實揭露」。
- 故**誠實延後**，待使用者決定投入。

## 但：共用槓桿的「機制」已就緒（零引擎成本可擴充）

統一站的設計讓「新增一科」幾乎不碰引擎：

1. **manifest 驅動**：`web/exams.js` 的 `EXAMS` 陣列加一筆即上架（key/prefix/name/subjects/notes）。
2. **按考試分檔**：放 `data/<新科>/bank.json`（＋可選 relations/explanations/essays）。
3. **共用題庫（真有重疊時）**：題庫 schema 可在題上加 `exams: ["a","b"]` 陣列標屬多科，
   loader 端依當前 `EXAM.key` 過濾 —— 一份資料餵多科、不重複存。現階段 5 科無重疊，未用到此欄。
4. landing 卡片由 manifest 自動生成（`exams.js` 對空的 `#exam-grid` 跑 `EXAMS.forEach`），零改動。

## 新增一科 SOP（未來接手）

1. **盤點考科**：用 Twinkle Hub `search_exam` / `get_exam_paper` 確認該考試的選擇題科目與
   資料可得性（question_count > 0 表示題目已結構化）。
2. **驗證重疊**：比對 stem 指紋，判斷能否共用既有題庫（多數情況不能，須自建）。
3. **建題庫**：沿用既有某科 `scripts/`（build_samp 系列）的 pipeline，產出
   `bank.json`（頂層 `{questions:[…]}` 物件，每筆題目欄位 qid/year/round/subject/no/stem/options/answer/parse），
   答案務必對齊官方標準答案 PDF。
4. **答案/品質**：低品質題標 `parse:"review"`（預設不出，可在學習藍圖勾選納入）。
5. **掛上統一站**：`data/<key>/` 放檔 + `exams.js` 加 manifest + landing 加卡片。**零引擎改動**。
6. **驗收**：本機 nocache server smoke —— 題庫載入、雷達、0 console error。
7. **若新科只考 1 科** → 先做下節「單科 robust 化」。

## 單科考試的 robust 化（觸發式待辦：上「只考 1 科」的證照前先做）

有些證照考試只考 1 科。引擎**核心**對單科已 guard、不會壞掉（`charts.js` 科數 <3 就不畫雷達、改提示看表格；`modes.js` launchMock 已特判單科、避免「全 1 科混合」並顯示科名）。但若真要上單科考試，下列**文案**在單科下語意會怪，需特判 —— 皆輕量條件文案、不動架構，來源為 2026-06-21 adversarial 紅隊 sim-single 壓測：

- `web/index.html` 雷達面板標題「能力雷達（N 科正確率）」→ 單科改「正確率」並隱藏雷達圖（面板已有 `radar-table` 可單獨保留）。
- `web/index.html` 模擬考標題「模擬考（混合出題）」→ 單科改「（隨機抽題）」或省略「混合」。
- `web/diagnostic.js` 結果頁「最該優先補強：X」與「各科能力雷達（起點）」h4 → 單科略過比較式／雷達字眼。
- `web/help.js` 使用說明「跨科混合」「各科正確率雷達」→ 單科改中性敘述。
- `web/modes.js` mock picker 引導文「單科／複數／全部…並選擇出題順序」→ 單科簡化並隱藏科目勾選區。
- `web/coach.js` 學習教練「把資源投到弱科／別平均分配」等跨科取捨金句 → 單科改走與科數無關的條目（間隔複習／主動作答等）。

**統一做法**：以 `SUBJECTS.length === 1` 分支，或在 `exams.js` 的 `renderExamChrome()` 依科數設定面板標題（比照雷達標題已用的 `data-exam-field` 機制）。

## 建議的下一個候選（依「資料現成度 × 重疊度」）
- 純文字、選擇題為主、資料結構化較好的類科優先（延續 5 科「無圖式裁切惡夢」的優勢）。
- 高考三級／普考的社會行政、人事行政等：選擇題比重需先查證（高考多申論，選擇題可能偏少）。
- 避免圖式重的類科（建築師、各類技師、醫學影像）——會撞上升學站正在處理的切圖難題。
